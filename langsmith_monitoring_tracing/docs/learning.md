# 📚 Learning Notes: LLM Observability & Production AI Systems

This document covers the key concepts behind this project. Good for anyone learning how to build and monitor production LLM applications.

---

## 🔍 LLM Observability and Monitoring

Traditional software is deterministic—if something breaks, logs and error traces are usually enough to identify the issue. LLM applications are different because they are non-deterministic; the same input can produce different outputs, and failures may originate from prompts, conversation history, retrieved context, or model behavior rather than the application code itself. This makes debugging significantly harder. LLM observability provides visibility into every model interaction, allowing developers to understand exactly what happened during a request and confidently operate AI systems in production.

### Why is it needed?

- **Debugging incorrect responses** — Inspect the exact prompt, context, and model output when users report bad answers.
- **Traceability** — Track the complete lifecycle of every LLM call, including prompts, responses, tokens, and latency.
- **Performance Monitoring** — Identify slow requests, latency spikes, and bottlenecks affecting user experience.
- **Cost Monitoring** — Track token usage and spending to prevent unexpected increases in LLM costs.
- **Failure Investigation** — Diagnose issues caused by prompts, context, model behavior, or API failures.
- **Production Reliability** — Detect silent failures and system degradation before they impact users.
- **Prompt Optimization** — Compare traces to understand which prompts produce better results.
- **Compliance & Auditing** — Maintain records of model interactions when traceability is required.

LLM Observability typically consists of three pillars:
- **Tracing** — Exact prompt, response, tokens, and metadata for every request.
- **Metrics** — Latency, cost, token usage, throughput, and error rates across the system.
- **Debugging** — Ability to replay and analyze historical requests to identify root causes.

## 📊 Monitoring — LangSmith + Custom Dashboard

### 🔍 LangSmith (Implemented in this project)

LangSmith is integrated via `LANGCHAIN_API_KEY` and `LANGCHAIN_TRACING_V2=true` in `.env`. Once enabled, every LLM call is automatically traced — LangChain handles it under the hood with no extra code needed.

LangSmith gives you two sections:

**Tracing tab** — A log of every individual LLM call. For each trace you can see:
- The **full prompt** sent to DeepSeek — system prompt, full conversation history, and the user's current message
- The **exact response** generated by the model
- **Token usage** — prompt tokens and completion tokens consumed
- **Latency** — how long the model took to respond
- **Thread grouping** — because we pass `session_id` into every LangChain call, all turns of a single conversation are grouped together, making it easy to follow a multi-turn chat end-to-end

**Monitoring tab** — Aggregate charts over time:
- Latency trends (average and percentiles)
- Token usage and cost over time
- Error rate and request volume

### 📈 Custom Dashboard (Built for learning)

In addition to LangSmith, this project includes a custom monitoring dashboard built from scratch. The goal was to understand what goes into building one — how to calculate latency percentiles, track cost per request, and maintain a rolling request log. It also covers application-level state that LangSmith does not, such as active sessions and server uptime. It is a learning exercise that complements LangSmith rather than replacing it.

## Real-time streaming

Instead of waiting for the full LLM response (which can take 5-10s), streaming yields tokens as they are generated.
Real-time response streaming was implemented using **Server-Sent Events (SSE)**. On the backend, FastAPI's `StreamingResponse` was combined with LangChain's asynchronous streaming interface (`astream`) to stream tokens as they are generated by the LLM. On the frontend, the native JavaScript `fetch` API and `TextDecoder` were used to process the incoming byte stream chunk-by-chunk and dynamically update the UI in real time, providing a more responsive user experience.

**Implementation using Server-Sent Events (SSE):**

```
Browser                         FastAPI                        DeepSeek API
  |                               |                                |
  |-- POST /chat/stream -------->|                                |
  |                               |-- astream() ----------------->|
  |                               |<-- chunk: "Hello" ------------|
  |<-- data: {"text": "Hello"} --|                                |
  |                               |<-- chunk: " there" ------------|
  |<-- data: {"text": " there"} -|                                |
  |                               |<-- [DONE] + usage_metadata ---|
  |<-- data: {"type":"metadata"} |                                |
```

**Key challenge:** Token usage and cost can only be calculated after the stream is complete. Solution: pass `stream_options={"include_usage": True}` — the final chunk contains usage metadata.

## 🏗 Architecture Decision: Why no Prometheus/Grafana?

**For this project**, the goal was a single-command setup. An in-memory Python metrics store + a `/api/metrics` JSON endpoint achieves the same observability (P99 latency, error rates, cost tracking) with zero external dependencies.

**Trade-off:** In-memory metrics reset on server restart. For production at scale, Prometheus + Grafana would be the right choice.

## 📈 Dashboard Metrics Reference

Every metric tracked in the built-in monitoring dashboard, with its definition.

### Overview
| Metric | Definition |
|---|---|
| **Uptime** | How long the server has been running since last start. |
| **Start Time** | Timestamp when the server was last started. |

### Request Counts
| Metric | Definition |
|---|---|
| **Total Requests** | Total number of chat requests received (both successful and failed). |
| **Successful Requests** | Number of requests that completed without error and returned a response. |
| **Failed Requests** | Number of requests that resulted in an error (API failure, timeout, etc.). |
| **Error Rate (%)** | `failed / total × 100` — percentage of all requests that failed. |
| **Error Breakdown** | Count of failures grouped by error type (e.g., `llm_error`, `timeout`, `unknown`). |

### Latency
| Metric | Definition |
|---|---|
| **Avg Latency** | Mean response time across all successful requests. |
| **P50 Latency (Median)** | 50% of requests completed faster than this value — the "typical" experience. |
| **P95 Latency** | 95% of requests completed faster than this — represents slow-but-not-worst cases. |
| **P99 Latency** | 99% of requests completed faster than this — represents near-worst-case performance. |
| **Min Latency** | Fastest response time ever recorded. |
| **Max Latency** | Slowest response time ever recorded. |

> **Why percentiles matter more than average:** If 99 requests take 1s and 1 request takes 60s, the average is 1.6s which looks fine. But P99 = 60s reveals that 1% of your users are having a terrible experience. Averages hide outliers; percentiles expose them.

### Token Usage
| Metric | Definition |
|---|---|
| **Total Prompt Tokens** | Cumulative tokens sent *to* the LLM (system prompt + history + user message). |
| **Total Completion Tokens** | Cumulative tokens *generated by* the LLM across all responses. |
| **Total Tokens** | Sum of prompt + completion tokens across all requests. |
| **Avg Tokens per Request** | Average total tokens consumed per successful chat request. |

### Cost
| Metric | Definition |
|---|---|
| **Total Cost (USD)** | Cumulative money spent on LLM API calls since server start. |
| **Avg Cost per Request** | `total_cost / successful_requests` — average spend per chat message. |

### Sessions & Activity
| Metric | Definition |
|---|---|
| **Active Sessions** | Number of unique conversation sessions currently held in memory. |
| **Recent Requests Log** | Rolling log of the last 100 requests with timestamp, session, latency, tokens, cost, and status. |
| **Request Timeline** | Per-minute bar chart showing request volume and errors over the last 30 minutes. |
